# 构筑系统与文案交互规范

## 当前构筑总框架

- 保留三栏结构：
  战斗栏 / 连携栏 / 技能栏

## 三栏职责

### 战斗栏

- 负责普攻和火力形态变化
- 目标是改变“这名角色怎么打、画面长什么样”

### 连携栏

- 负责切换角色入场、退场、后台协同、轮转收益
- 目标是让切人产生明确节奏收益

### 技能栏

- 负责大招与技能类强化
- 目标是强化大招价值、节奏控制和战术爆发

## 当前构筑问题与共识

- 早期做法是三名角色同时吃到同一套 Build
- 优点：三人都能一起变强，不会只强化到一个人
- 缺点：角色之间容易同质化，看起来只是描述不同，本质效果一致

当前共识：

- 三个人都必须值得切
- 三个人都不残废
- 三个人都能有高光
- 但三个人不是同一种高光

## Build 菜单交互规范

### 一级菜单

- 只显示分类名：
  战斗 / 连携 / 技能

### 二级菜单

- 只显示当前站场角色对应的效果描述
- 不显示另外两名角色的效果
- 支持短按直接选择
- 支持长按或悬停进入三级菜单

### 三级菜单

- 显示完整详细效果
- 可展示其他角色对应效果
- 可直接点击卡片选择
- 不需要额外“选择这个”按钮

## 最终效果卡规范

- 二级菜单右上角只显示主名，例如“天灾化身”
- 进入详情后显示完整名：
  `天灾化身：断神诸恶`
- 详情里只展示需求卡与等级进度
- 已达成的需求高亮，未达成的灰显
- 不写“已激活/未激活”

## 文案硬规则

- 禁止设计师黑话
- 禁止只写空泛气氛词
- 禁止引入新的抽象机制名词去增加理解成本
- 直接写“玩家做什么，会发生什么”

## 文案统一写法

- “开大”统一写成“释放大招”
- 百分比伤害统一写成：
  `伤害为原来的 XX%`
- 二级菜单写“简述效果”
- 三级菜单写“完整效果 + 进化要求”

## 当前构筑方向共识

- 构筑不该只提供数值
- 应该优先提供：
  战斗方式变化
  视觉表现变化
  轮转收益变化
  最终进化效果

## 当前卡池设计注意事项

- 不允许为了凑数而生成重复卡
- 不允许名字很华丽但实际效果重复
- 最终效果卡必须在视觉上有明显辨识度
- 如果卡的收益玩家看不见，优先补演出
